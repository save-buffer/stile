- Consider adding a `Masks` class that automatically broadcasts up to the desired shape

- Static lint pass for common error-prone kernel patterns. Cheaper +
  more directly useful than full AA-bound quantization analysis, and
  more discoverable for kernel authors. Candidate patterns:
    * Reduction without explicit `keepdims` or axis (silent broadcast
      surprises).
    * Pointwise op on tensors with the same shape but mismatched dim
      names (verifier would catch this, but lint can flag before
      verification).
    * `tl.load` of a tile that exceeds the dim's `size` without a
      mask — masks aren't required by stile but are usually the
      author's intent.
    * `exp(x)` immediately followed by `sum(...)` without a stable-
      softmax max-subtract (overflow risk in fp16 / fp8).
    * Accumulator declared at the same dtype as the operand (no
      upcast) — flagged because it's almost always a bug in practice.
    * Triton kernel body that allocates a `tl.zeros` tile of size
      `> 4 * native_tile_size` (register-pressure warning).
  Implementation shape: walk the typed-ET / kernel body in
  `stile/triton/_core.py` post-verification, accumulate findings,
  surface as a `lint(kernel_fn)` API or auto-print during `verify`.

- Quantization-aware AA / first-class `quantize(x, scheme=...)` op.
  Back-burner — see memory `project-aa-quantization`. The
  `astype(dtype_str)` path captures IEEE rounding but not the
  calibration-dependent error from real quantizers (per-tensor /
  per-channel / per-tile fp8). Sensitivity-ranking is good enough for
  now; revisit when a concrete user need surfaces.

- DONE (2026-05-28): KV-writeback. Generalized `Scatter` to carry an
  explicit `base` (the value at untouched positions; defaults to 0 =
  old "scatter into zeros"). Writeback is `base except [dim @ idx] =
  values` (TLA+-style surface syntax) or `scatter[dim](values, idx,
  base)` (function form), both lowering to `Scatter(source=values,
  dim_in_dest=dim, idx=idx, base=base)`. Verifier keeps scatter opaque;
  base is part of its identity (writeback ≠ zero-scatter). Permutation
  round-trip ignores base (full overwrite). See test_scatter_writeback.py.
  NOTE: only jax backend has a runtime `.scatter()`; torch/numpy would
  need one if a kernel test targets them.

- Coverage check generalization to coupled axes. Today the check is
  per-axis: union the interval set along each dim independently and
  compare to `[0, dim.size)`. Works when stores write axis-aligned
  rectangles whose axes vary independently — the typical pattern.
  Fails to verify (false positives) when different (pid_m, pid_n)
  combinations write different sub-rectangles in a diagonal pattern.
  Generalization would need true N-D interval-union math.

- `tl.reshape` is intentionally rejected by the typed-Triton verifier
  (`stile/triton/_core.py` raises on encounter). Reasoning:

    Reshape's two real use cases — singleton squeeze/unsqueeze and
    flatten/unflatten of contiguous dims — both have alternatives that
    fit the stile type system without new infrastructure:

      - Axis permute / transpose      → `tl.trans(x)` (already verified).
      - Insert / squeeze singleton    → `x[:, None]` (broadcast handler
        already handles), or implicit via stile's `_broadcast_pair`.
      - Sub-tile selection            → `ttl.load(ptr, DIM[lo:hi], …)`
        with the typed slice syntax.

    True flatten/unflatten would require a "compound dim" type in
    stile so that e.g. `(BM*BN,)` had a recoverable identity as the
    row-major flatten of `(BM, BN)`. The normalizer would then need
    a rule like
        `reduce(compound, op, body) ≡ reduce(d1, op, reduce(d2, op, body))`
    so a `tl.sum(reshape(x, (BM*BN,)))` matched `sum_BM sum_BN X`.
    That's a substantial extension — not on the path right now.

    If a use case appears that genuinely can't be expressed via
    rearrange/trans/slice/broadcast, the path forward is to land
    compound-dim infrastructure rather than weakening the verifier.
